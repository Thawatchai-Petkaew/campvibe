1 ปัญหา + หลักการ

ทำไมต้องมีมาตรฐาน Loading UX และเราตัดสินใจอะไร

ปัญหาที่พบ (Before)

CampVibe เคยใช้ skeleton เดียวกัน (catalog-grid) กับทุก route — ผู้ใช้ที่เข้า /dashboard หรือ /profile เห็น grid ของ camp-card skeleton แทนที่จะเป็น layout ของหน้านั้น ๆ ผลคือ layout shift เมื่อ content จริงโหลด (CLS > 0) และ skeleton "กะพริบ" หายไปเร็วมากเมื่อ load ใน <300ms — อ่านเป็น glitch ไม่ใช่ feedback

หลักการที่เราใช้ (After)

1
Skeleton mirrors the layout exactly
skeleton card = card จริงทุก dimension (aspect-ratio / border-radius / grid / gap) → CLS = 0
2
Section-level default — ไม่ blank ทั้งหน้า
Navbar + filter bar + ทุก chrome render ทันที — <Suspense> ครอบเฉพาะส่วน async
3
Right loader per wait-time (decision matrix)
<300ms = ไม่แสดง skeleton / >1s + รู้ layout = skeleton / button action = inline spinner
4
Anti-flicker pair: delay-before-show + min-display
กันการ flash ที่อ่านเป็น glitch ทั้งคู่ต้องทำงานพร้อมกัน

2 Decision Matrix — เลือก Loader อะไร สำหรับ scenario ไหน

ใช้ตารางนี้ก่อนเสมอ — 1 loading region = 1 loader เท่านั้น ห้าม stack ซ้อนกัน

Scenario / Wait time Loader แสดงหลัง หมายเหตุ
< ~100ms (fast) ไม่แสดง loader — NN/g: <100ms = instant ในมุมมองผู้ใช้ — loader จะรบกวนมากกว่า
เช่น: filter ที่ cache warm, navigation ที่ shell render ทันที
~100ms–1s · ไม่รู้ layout Delayed Spinner หรือไม่แสดง ~300ms Delay กัน flash — ถ้า resolve ก่อน 300ms ผู้ใช้ไม่เห็น loader เลย
> ~1s · navigation / รู้ layout
(list, cards, detail)		 Skeleton ตรงกับ route นั้น ทันที กฎหลัก: skeleton ต้อง mirror layout จริงทุก dimension → CLS = 0
Home: CampgroundGridSkeleton · Dashboard: DashboardOverviewSkeleton
> ~1s · isolated widget / module Spinner (LoadingSpinner) ทันที ใช้เมื่อ widget เล็ก ไม่รู้ layout context
User action · form submit · button Inline spinner บน control ทันที Feedback อยู่ที่ปุ่มที่ผู้ใช้กด — ไม่ใช่ page overlay หรือ global loader
ตัวอย่าง: ปุ่ม "บันทึก" → spinner ใน button ขณะ submit
Mutation คาดผลได้ (like, toggle, add) Optimistic UI ทันที Apply state ที่คาดไว้ก่อน reconcile/rollback เมื่อได้ response
เช่น: wishlist toggle — ไอคอนเปลี่ยนทันที ไม่รอ API
> ~10s หรือรู้ progress (upload, export) Progress bar + cancel ทันที Nielsen: >10s ต้องมี % เสร็จ + ทางออก — bare spinner คือการทิ้งผู้ใช้ไว้
Background refetch ขณะผู้ใช้อ่านอยู่ Subtle indicator (inline) — ห้าม blank content ที่ผู้ใช้กำลังอ่าน — แสดง indicator เล็ก ๆ ตรงมุม

3 User Journey — สิ่งที่ผู้ใช้เห็นในแต่ละ scenario

5 เส้นทาง ครอบคลุม loading scenarios หลักของ CampVibe พร้อมเหตุผลจาก matrix

A — Cold-load Home (เปิด URL ครั้งแรก)

1
ผู้ใช้พิมพ์ URL campvibe.app ใน browser
Request ส่งไป server — browser ยังว่างอยู่
2
Navbar + CategoryBar + FilterBar paint ทันที
Static chrome render พร้อม HTML จาก server — ผู้ใช้เห็น logo, ช่องค้นหา, category icons ก่อนเลย ไม่รอ data
Chrome: solid ทำไม: Navbar/CategoryBar/FilterBar คือ static component — ไม่มี async dependency
3
ถ้า data ไม่มาใน 300ms — Grid skeleton ปรากฏ
CampgroundGridSkeleton โผล่หลัง 300ms (CSS animation-delay) — แสดง 5-column grid ของ card-skeleton ที่มีขนาดเท่า card จริงทุก pixel: aspect-square image + 3 text bars
Loader: Skeleton (section-level) ทำไม: matrix row 3 — >1s + รู้ layout → skeleton per route Grid: 5-col (xl) / 4-col (lg) / 3-col (md) / 2-col (sm) / 1-col
4
ถ้า data มาใน <300ms — ไม่มี skeleton เลย
Card จริงโชว์ตรง — ไม่มี flash ไม่มี skeleton กะพริบ ผู้ใช้ไม่รู้ว่ามี loading UI อยู่
Fast path ทำไม: delay-before-show ~300ms กัน flash
5
Data มา — skeleton swap กับ card จริง (zero CLS)
เพราะ skeleton มีขนาดเท่า card จริงทุก dimension — layout ไม่ขยับเลยเมื่อ card โหลดเสร็จ
CLS = 0 ทำไม: skeleton mirrors layout exactly (กฎหลักของ standard)

B — เปลี่ยน Filter หรือ Sort (Section-level re-skeleton)

1
ผู้ใช้เลือก category / province / sort
Navbar + CategoryBar + FilterBar ยังอยู่ครบ — ไม่หายไปไหน
Shell stays intact
2
เฉพาะ grid area แสดง skeleton ทันที
Suspense key เปลี่ยน → fallback (CampgroundGridSkeleton) โชว์ทันที โดย shell ทั้งหมดยังแสดงอยู่ ผู้ใช้รู้ว่ากำลัง filter อยู่
Loader: Section-level Skeleton ทำไม: <Suspense key={searchParamsKey}> ใน app/page.tsx
3
ผลลัพธ์ใหม่มา — swap เข้า grid
ผู้ใช้เห็น camp ที่ filter แล้ว — shell ไม่เคยหายไป ประสบการณ์ลื่นไหล
Shell never blanked

C — Navigate ไป /dashboard (Route-level skeleton)

1
ผู้ใช้คลิก "Dashboard" ใน Navbar dropdown
Next.js App Router เริ่ม load route ใหม่
2
DashboardOverviewSkeleton แสดงทันที
Chrome (Navbar) ยังอยู่ — ส่วน content area แสดง skeleton ที่ตรงกับ dashboard: header row + 3 stat cards + table rows (ไม่ใช่ camp-card grid)
Loader: DashboardOverviewSkeleton (route-specific) ทำไม: dashboard มี layout ต่างจาก Home ต้องใช้ skeleton ของตัวเอง
3
API response มา — real stats + table โชว์
Skeleton หายไป แสดงตัวเลขจริง + booking list ไม่มี layout shift
Route-specific skeleton = zero CLS

D — Button / Action (Inline spinner)

1
ผู้ใช้กดปุ่ม "ยืนยันการจอง" หรือ "บันทึก"
Form submit ส่งข้อมูลไป API
2
Inline spinner โผล่ใน button ทันที — ปุ่ม disabled
หน้าจอไม่เปลี่ยน — feedback อยู่ที่ปุ่มที่ผู้ใช้เพิ่งกด ไม่มี page overlay ไม่มี skeleton ไม่มีการ blank หน้า
Loader: Inline spinner (LoadingSpinner) ทำไม: matrix row 5 — user action feedback = local ที่ control
3
Response มา — spinner หาย, success/error state แสดง
สำเร็จ: toast + redirect · ล้มเหลว: ErrorBanner + ปุ่มกลับมา enabled
Local feedback = clear causality

E — Wishlist Toggle (Optimistic UI)

1
ผู้ใช้กดไอคอน heart บน camp card
Mutation ส่งไป API — แต่เราคาดผลได้ (toggle)
2
ไอคอนเปลี่ยนสีทันที — ไม่รอ API
ใช้ useOptimistic — state เปลี่ยนทันทีในหน้าจอ ไม่มี spinner ไม่มี delay ที่ผู้ใช้รู้สึก
Loader: Optimistic UI (useOptimistic) ทำไม: matrix row 6 — mutation ที่คาดผลได้ = optimistic ทันที
3
API ตอบกลับ — reconcile หรือ rollback
สำเร็จ: state ยืนยัน ไม่เปลี่ยน · ล้มเหลว: rollback กลับสถานะเดิม + toast error
Zero perceived latency

5 Accessibility (WCAG 2.1 AA) — สำหรับ Loading States

Loading UI ที่ขาด a11y เป็นหน้าจอที่ผู้ใช้ screen reader เห็นแค่ความเงียบ

A
aria-busy + role="status" + aria-live="polite"
ใส่บน loading region ทั้งก้อน — aria-busy="true" ขณะโหลด / aria-busy="false" เมื่อเสร็จ · role="status" aria-live="polite" ทำให้ screen reader ประกาศ label โดยไม่ขัดสิ่งที่กำลังอ่านอยู่
ก
Thai label: "กำลังโหลด…"
text node sr-only ภายใน live region — อ่านครั้งเดียวต่อ mount · ไม่ใช่ aria-label บน card shape แต่ละใบ (= noise) · ตัวอย่างใน CampgroundGridSkeleton: <span class="sr-only">{SR_LABEL}</span>
👁
aria-hidden="true" บน skeleton shapes
shape แต่ละชิ้น (image box, text bar) คือ decorative — aria-hidden="true" กัน screen reader อ่านซ้ำไม่มีความหมาย · CampgroundSkeleton card ทุกใบมี aria-hidden="true" บน wrapper
✦
prefers-reduced-motion — ปิด shimmer animation
ผู้ใช้ที่ตั้ง reduced-motion จะไม่เห็น shimmer sweep — ใช้ static bg-muted แทน
@media (prefers-reduced-motion: reduce) { .shimmer { animation: none; } }
◉
Focus management (skeleton → content)
เมื่อ skeleton swap เป็น content — focus ไม่ควรหายไป (ถ้าผู้ใช้ navigate ด้วย keyboard) · ในทาง Next.js App Router Suspense: focus ยังอยู่ที่ element เดิมเมื่อ content เข้า
🎨
Token: bg-muted เท่านั้น — ห้าม bg-gray-*
skeleton fill ใช้ bg-muted (token) เสมอ — dark mode flip ผ่าน .dark อัตโนมัติ · bg-gray-200 fails npm run check:palette → CI exit 1

6 Pilot Notes — สิ่งที่เรียนรู้จาก Home (LOAD-1)

Home คือ reference implementation — pilot findings นี้เป็นฐานก่อน rollout ไป S2–S4

Home เป็น reference ที่สำเร็จแล้ว

CAM-197 (LOAD-1) สร้าง CampgroundGridSkeleton + Suspense boundary ใน app/page.tsx ด้วย key={searchParamsKey} ทำให้ filter/sort ทุกครั้ง trigger skeleton ทันที (ไม่ต้อง re-mount page, แค่ key เปลี่ยน → fallback โชว์)

Anti-flicker: CSS animation-delay ใช้ได้ดีบน Suspense fallback

skeleton-delay-show class (opacity 0 → 1 หลัง 300ms) ใน globals.css ทำ delay-before-show สำหรับ Suspense fallback ได้ผลดี — load ที่เร็วกว่า 300ms skeleton ไม่เคย paint ให้เห็น → zero flash

Min-display บน Suspense fallback: friction สูง

Suspense ของ React dismiss fallback ทันทีที่ child resolve — ไม่มี hook ให้ delay dismiss การทำ min-display บน Suspense fallback ต้องใช้ workaround ที่ซับซ้อน (setTimeout + state toggle) และ pilot ตัดสินใจไม่ implement สำหรับ Suspense path ในขณะนี้

useMinimumLoading hook — สำหรับ client-fetch component (Dashboard)

Dashboard ใช้ useEffect + fetch (client component) — ที่นี่ min-display ทำได้ง่ายกว่า: เก็บ startTime ตอน fetch เริ่ม → เมื่อ response มา ถ้าเวลาผ่านไปน้อยกว่า 300ms ให้ setTimeout ก่อนเปลี่ยน state จาก loading → done
→ นี่คือ pattern ที่จะ extract เป็น useMinimumLoading hook ก่อน S2–S4

Rollout plan (S2–S4)

S2 — RootShellSkeleton (minimal last-resort สำหรับ app/loading.tsx) · S3 — ProfileFormSkeleton (profile form section, section-level) · S4 — BookingListSkeleton (booking list section, section-level)
ทุก story ใช้ standard นี้เป็นสเปก + ผ่าน Design Gate §6 ก่อน merge
