## 1. Schema change

### Fields added to `CampSite`

```prisma
// AGG-1 (CAM-189): denormalized aggregate — maintained on every review write inside a
// transaction; recomputed by reconcile job as backstop. Derivation trail:
//   avgRating = roundAvgRating(computeAvgRating(non-deleted Review.rating for this camp))
//   reviewCount = COUNT of non-deleted Review rows for this camp
// Source: lib/review-summary.ts:roundAvgRating + lib/sort-utils.ts:computeAvgRating
avgRating   Decimal? @db.Decimal(2, 1) // [Public] null when reviewCount = 0
reviewCount Int      @default(0)       // [Public]
```

**Classification:** both fields are Public (no PII, no Financial, no Geo).

**Placement:** insert after the existing `version Int @default(1)` field and before the `// Relations` block (schema line ~316, immediately before `locationId`).

**Index for PERF-5 (future DB-sort):** the rating-sort index is deferred to PERF-5. This story adds only the columns. The PERF-2 composite indexes already on the model (`@@index([isPublished, deletedAt, priceLow, id])` and `@@index([isPublished, deletedAt, createdAt, id])`) are not touched.

### Why `Decimal(2,1)` and not `Float`

`roundAvgRating` (lib/review-summary.ts:26) returns `Math.round(avg * 10) / 10` — always a single decimal place (e.g. 4.3). Storing as `Decimal(2,1)` guarantees exact decimal representation and matches the convention established in ADR-002 (money = Decimal, not Float). `Float` would introduce a binary rounding artifact (e.g. 4.300000000001). `Decimal(2,1)` has the range 0.0–9.9, which fully covers the Review.rating domain of 1–5. A full rating avg is at most 5.0, so the range is sound.

**Open question for owner — see §8.**

## 2. Migration plan

### Migration type: additive + backfill

The migration is purely additive (ADD COLUMN). No existing column is renamed, typed-changed, or dropped. Reversible via a down migration that DROPs the two columns.

### Migration SQL shape

Prisma generates the `ALTER TABLE` automatically from the schema diff. The backfill is a custom SQL block appended directly into the generated `migration.sql` (Prisma supports appending raw SQL after the generated DDL). The backfill runs once at migration time and is idempotent — running it again on already-backfilled rows produces the same result (no rows change).

```sql
-- Prisma-generated (example — actual generated by `prisma migrate dev --create-only`)
ALTER TABLE "CampSite"
  ADD COLUMN "avgRating" DECIMAL(2,1),
  ADD COLUMN "reviewCount" INTEGER NOT NULL DEFAULT 0;

-- AGG-1 backfill: compute from existing non-deleted reviews.
-- Runs once; idempotent (re-running produces same result).
UPDATE "CampSite" cs
SET
  "reviewCount" = COALESCE(agg."cnt", 0),
  "avgRating"   = CASE WHEN COALESCE(agg."cnt", 0) = 0 THEN NULL
                       ELSE ROUND(agg."avg_rating"::NUMERIC, 1)
                  END
FROM (
  SELECT
    r."campSiteId",
    COUNT(*)::INT                AS "cnt",
    AVG(r."rating"::NUMERIC)     AS "avg_rating"
  FROM "Review" r
  WHERE r."deletedAt" IS NULL
  GROUP BY r."campSiteId"
) AS agg
WHERE cs."id" = agg."campSiteId";
-- Camps with zero reviews keep reviewCount=0 and avgRating=NULL (DEFAULT already set).
```

**Derivation match:** `ROUND(AVG(rating), 1)` in SQL mirrors `roundAvgRating(computeAvgRating(reviews))` in TypeScript:
- `computeAvgRating` (lib/sort-utils.ts:18) — `sum/length`, null when empty.
- `roundAvgRating` (lib/review-summary.ts:26) — `Math.round(avg * 10) / 10`, null when avg is null.
- `ROUND(x::NUMERIC, 1)` in Postgres is equivalent. Both use round-half-up (Postgres default).

**Reversibility (down migration):**

```sql
ALTER TABLE "CampSite"
  DROP COLUMN "avgRating",
  DROP COLUMN "reviewCount";
```

The down drops both columns; no data is corrupted; the app falls back to computing on-the-fly.

**Deployment path:** `prisma migrate deploy` in the Vercel build script applies the migration automatically per the existing 3-env flow (Local → Staging → Production). The backfill SQL runs as part of the same migration transaction — atomic with the column add.

**Risk — large tables:** if the production `Review` table is large, the UPDATE scans all non-deleted reviews. For the current catalog size (not measured), this is expected to be fast. If the table grows significantly before this ships, run the backfill in batches; however, at current scale a single UPDATE is the simpler and correct choice. Test on Staging first and measure duration before promoting to prod.

## 3. Maintenance mechanism

### Recommendation: recompute inside the same `$transaction` as each review write

The primary mechanism is an in-transaction recompute on every review write. This keeps the aggregate consistent without relying on a Postgres trigger (which is opaque to TypeScript type-checking and Prisma drift detection) and without a separate async job that could miss writes.

**Why not a Postgres trigger?**

The `Review` model uses soft-delete (`deletedAt DateTime?`). A soft-delete is an `UPDATE`, not a `DELETE` — a standard `AFTER DELETE` trigger does not fire. A trigger on UPDATE that checks `deletedAt` is possible but introduces schema drift, is not type-checked, and is invisible in Prisma's generated client. The ADR index (ADR-000) already records the convention that behavioral logic lives in the service/API layer. Trigger-as-primary is therefore rejected.

### Pattern for `POST /api/reviews` (app/api/reviews/route.ts)

Current code (route.ts:54–63) calls `prisma.review.create(...)` directly. Replace this single call with a `$transaction` that creates the review and then recomputes the aggregate for that camp:

```ts
// lib/prisma.ts — singleton import: import { prisma } from '@/lib/prisma'

const [review] = await prisma.$transaction([
  // Step 1: create the review (unchanged from current)
  prisma.review.create({
    data: {
      authorId,
      campSiteId: data.campSiteId,
      rating: data.rating,
      title: data.title ?? '',
      content: data.content,
      visitDate: data.visitDate ? new Date(data.visitDate) : null,
    },
  }),
  // Step 2: recompute avgRating + reviewCount for this camp from the source rows.
  // Uses a raw update-from-subquery so Prisma stays the only DB access layer.
  // The aggregate() call is the cleanest Prisma-native approach:
  prisma.campSite.update({
    where: { id: data.campSiteId },
    data: {
      reviewCount: await prisma.review.count({
        where: { campSiteId: data.campSiteId, deletedAt: null },
      }),
      avgRating:
        (await prisma.review.aggregate({
          where: { campSiteId: data.campSiteId, deletedAt: null },
          _avg: { rating: true },
        }))._avg.rating != null
          ? new Prisma.Decimal(
              Math.round(
                Number(
                  (await prisma.review.aggregate({
                    where: { campSiteId: data.campSiteId, deletedAt: null },
                    _avg: { rating: true },
                  }))._avg.rating!
                ) * 10
              ) / 10
            )
          : null,
    },
  }),
]);
```

**Implementation note for backend:** the pattern above is illustrative; the actual implementation should call `prisma.review.aggregate` and `prisma.review.count` once each, capturing results into local variables before the `$transaction` block, then use an interactive transaction (`prisma.$transaction(async (tx) => { ... })`) so the count and aggregate queries execute inside the same transaction as the create and update. The interactive-transaction form is preferred here because the aggregate query result must be computed before the update is issued:

```ts
// Preferred interactive-transaction form
const review = await prisma.$transaction(async (tx) => {
  const created = await tx.review.create({ data: { ... } });

  const agg = await tx.review.aggregate({
    where: { campSiteId: data.campSiteId, deletedAt: null },
    _avg: { rating: true },
    _count: { id: true },
  });

  const count = agg._count.id;
  const rawAvg = agg._avg.rating;
  const avgRating =
    rawAvg != null
      ? new Prisma.Decimal(Math.round(Number(rawAvg) * 10) / 10)
      : null;

  await tx.campSite.update({
    where: { id: data.campSiteId },
    data: { reviewCount: count, avgRating },
  });

  return created;
});
```

This reads the aggregate AFTER the `review.create` (so the new row is included), then writes the result to `campSite` — all in one atomic operation.

### Contract for future PATCH / DELETE review routes

**PATCH and DELETE review routes do not exist yet** (confirmed: `app/api/reviews/` contains only `route.ts` with POST). When those routes are built, they MUST apply the same maintenance pattern: wrap the review mutation (update or soft-delete) in a `$transaction` that recomputes `avgRating` and `reviewCount` for the affected `campSiteId` immediately after. This is a build-time contract for those future stories; the reconcile job covers drift in the interim.

## 4. Reconcile job

### Purpose

A self-healing backstop. If any code path writes a review without updating the aggregate (e.g. a seed script, a future PATCH route before it's wired up, or a transaction that was partially applied), the reconcile job corrects all camps in one pass.

### Location

`scripts/reconcile-ratings.mjs`

### Logic (pseudocode)

```js
// scripts/reconcile-ratings.mjs
import { PrismaClient } from '@prisma/client';
const prisma = new PrismaClient();

async function main() {
  // Recompute for every camp that has at least one review OR a non-zero stored count.
  // Using a raw query mirrors the backfill SQL exactly — same derivation trail.
  await prisma.$executeRaw`
    UPDATE "CampSite" cs
    SET
      "reviewCount" = COALESCE(agg."cnt", 0),
      "avgRating"   = CASE WHEN COALESCE(agg."cnt", 0) = 0 THEN NULL
                           ELSE ROUND(agg."avg_rating"::NUMERIC, 1)
                      END,
      "updatedAt"   = NOW()
    FROM (
      SELECT
        r."campSiteId",
        COUNT(*)::INT                AS "cnt",
        AVG(r."rating"::NUMERIC)     AS "avg_rating"
      FROM "Review" r
      WHERE r."deletedAt" IS NULL
      GROUP BY r."campSiteId"
    ) AS agg
    WHERE cs."id" = agg."campSiteId"
  `;

  // Also zero out camps that have no reviews but a stale reviewCount > 0.
  await prisma.$executeRaw`
    UPDATE "CampSite"
    SET "reviewCount" = 0, "avgRating" = NULL, "updatedAt" = NOW()
    WHERE "id" NOT IN (
      SELECT DISTINCT "campSiteId" FROM "Review" WHERE "deletedAt" IS NULL
    )
    AND "reviewCount" > 0
  `;

  console.log(JSON.stringify({ level: 'info', event: 'reconcile_ratings_complete' }));
}

main().finally(() => prisma.$disconnect());
```

### Invocation

```json
// package.json — add to scripts
"reconcile:ratings": "node scripts/reconcile-ratings.mjs"
```

Run manually: `npm run reconcile:ratings` (with `DATABASE_URL` set for the target env).

Cron wiring is deferred — not in scope for this story. A nightly/weekly cron can be added later as a follow-up. The job is idempotent: running it multiple times in a row produces the same result.

### Idempotency

Both UPDATEs are self-contained full-table recomputes. Running twice changes no row a second time if the source data has not changed.

## 5. Consistency contract (derivation trail — architecture.md §12)

```
CampSite.avgRating  = roundAvgRating(computeAvgRating(Review[campSiteId=X, deletedAt=NULL].rating))
                    = ROUND(AVG(Review.rating WHERE campSiteId=X AND deletedAt IS NULL), 1)
                    = NULL when reviewCount = 0

CampSite.reviewCount = COUNT(Review WHERE campSiteId=X AND deletedAt IS NULL)
```

**Source Pixels:** `Review.rating` (Int 1–5, lib/sort-utils.ts:computeAvgRating) + `Review.deletedAt` (soft-delete sentinel).

**The column is a cache, not the source of truth.** The source of truth is always the `Review` table. The column is reproducible from `Review` at any time via the reconcile job. This satisfies architecture.md §12: the cached aggregate names its source Pixels, is reproducible, and is not the only place the fact lives.

**null contract:** `avgRating` is nullable. It MUST be NULL when `reviewCount = 0`. A camp with zero reviews must not store 0.0 — null conveys "no data" distinctly from a real zero average, and it matches `computeAvgRating` returning null for an empty array (lib/sort-utils.ts:19).

## 6. Scope boundary

**This story (CAM-189) delivers:**

1. Schema: `avgRating Decimal(2,1)?` + `reviewCount Int @default(0)` on `CampSite`.
2. Migration SQL: additive ALTER TABLE + one-time backfill UPDATE (reversible).
3. Maintenance: `POST /api/reviews` route.ts wrapped in interactive `$transaction` that recomputes aggregate after each review create.
4. Reconcile job: `scripts/reconcile-ratings.mjs` + `npm run reconcile:ratings`.

**This story does NOT deliver (explicit out-of-scope):**

- **PERF-5 (DB rating sort):** The `@@index([isPublished, deletedAt, avgRating, id])` index and the Prisma `orderBy: { avgRating: 'desc' }` path in `app/page.tsx` are PERF-5's work. The in-memory `sortByRating` in lib/sort-utils.ts and the scale guard comment remain in place until PERF-5 ships.
- **PERF-1 (drop reviews from campCardSelect):** `lib/read-models/camp-card.ts` still includes `reviews: { where: { deletedAt: null }, select: { rating: true } }`. Removing it and reading `avgRating`/`reviewCount` columns directly is PERF-1's work.
- **PATCH / DELETE review routes:** do not exist yet; they will maintain the aggregate when built. Reconcile is the backstop in the interim.

The build agent MUST NOT touch `app/page.tsx` sort logic, `lib/read-models/camp-card.ts`, or `lib/sort-utils.ts` as part of this story.

### QA targets

| Scenario | Expected result | How to verify |
|---|---|---|
| Backfill: existing camp with 3 reviews (ratings 4, 5, 3) | `reviewCount=3`, `avgRating=4.0` (avg=4.0, rounds to 4.0) | Run migration on staging; SELECT and compare |
| Backfill: camp with 0 reviews | `reviewCount=0`, `avgRating=NULL` | SELECT on zero-review camp post-migration |

| POST /api/reviews: create second review (rating=4) on same camp | `reviewCount` = 2; `avgRating` = 4.5 | POST then SELECT |
| Reconcile: manually corrupt a camp's reviewCount to 999 | After `npm run reconcile:ratings`: value corrected to truth | UPDATE directly in DB, run script, SELECT |
| Reconcile: camp with 0 reviews but reviewCount=5 (stale) | After reconcile: `reviewCount=0`, `avgRating=NULL` | Same pattern |
| Rounding: avg = 4.35 (e.g. ratings 4, 4, 5 = 4.333…) | `avgRating=4.3` (round half-up at 1 decimal) | Verified by roundAvgRating unit test in lib/review-summary.test.ts |

### Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Backfill slow on large Review table | Low (current catalog is small; not measured) | Measure duration on Staging before prod promote; if >30s, add batching |
| Transaction contention on review-create | Low (review creates are infrequent relative to reads) | Interactive transaction is short (one aggregate + one update); contention unlikely |
| Future review PATCH/DELETE route forgets to maintain aggregate | Medium (future story) | Reconcile job catches drift; the contract is documented in §3 |
| `Decimal(2,1)` precision insufficient | Not a risk — max avg is 5.0, range 0.0–9.9 covers all cases | N/A |
| Migration auto-runs on Vercel build and backfill fails mid-run | Low | The backfill is a single UPDATE wrapped in the migration transaction; it either fully succeeds or fully rolls back |

## 8. Open questions for owner (G2)

**Q1 — Decimal(2,1) vs Float (type decision):**

- Option A: `Decimal(2,1)` — exact representation; consistent with ADR-002 (money = Decimal, not Float); columns like `vatRate Decimal @db.Decimal(5,4)` already use this pattern; recommended.
- Option B: `Float` — simpler Prisma type; no Decimal import needed; slightly less correct (binary rounding artifact e.g. `4.3000000001`); inconsistent with ADR-002 convention.

This is the only non-obvious type choice. Recommended: Decimal(2,1). Confirm or override.

**Q2 — Reconcile invocation schedule (informational, no decision blocks the build):**

The script ships as `npm run reconcile:ratings` (manual). A nightly/weekly cron (Vercel Cron Job or an external scheduler) can be added in a follow-up. No decision needed to unblock CAM-189.
